1. Pipeline 整体架构和各环节职责划分

本项目是一个端到端的动画元数据增量处理 Pipeline，整体采用五阶段串行架构：

    Fetch → Quality Check → Match → Export → Report

各环节职责如下：

(1) Fetch（数据获取 — fetcher.py）
    职责：从外部数据源获取 MAL 动画元数据并写入 SQLite 数据库。
    实现：支持两种数据源模式——
      - API 模式：调用 Jikan REST API（api.jikan.moe/v4），按页获取，每页 25 条，
        内置 1.2 秒间隔限流和指数退避重试（最多 5 次），处理 429 限流响应。
      - CSV 模式：从本地 Kaggle CSV 文件批量加载，自动检测分隔符（csv.Sniffer），
        通过 CSV_COLUMN_MAP 字典将不同数据集的列名统一映射为内部字段名，
        每 500 条一批写入数据库并更新 checkpoint。
    增量机制：API 模式记录页码，CSV 模式记录"文件名:行号"，下次运行从断点继续。

(2) Quality Check（质量检查 — quality.py）
    职责：对已入库的动画数据逐条执行质量规则检查，筛选出合格数据进入匹配环节。
    实现：定义了 4 条规则——
      - required_fields：mal_id 和 title 必须存在且非空
      - score_range：score 若存在须在 [0, 10] 范围内
      - year_validity：year 若存在须在 [1900, 2030] 范围内
      - episodes_non_negative：episodes 若存在须 >= 0
    不合格数据写入 quality_issues 表（记录 mal_id、规则名、问题描述），不阻塞流程。
    通过质量检查的记录才会进入后续匹配阶段。

(3) Match（跨源匹配 — matcher.py）
    职责：将通过质量检查的 MAL 条目与 Wikidata 知识库中的实体进行跨源匹配。
    实现：分两步——
      - Step 1 精确 ID 匹配：通过 SPARQL 查询 Wikidata 的 P4086 属性（MAL anime ID），
        每批 50 个 ID 构造 VALUES 子句批量查询，匹配到的置信度为 1.0。
      - Step 2 模糊标题匹配：对精确未匹配到的记录，从 Wikidata 获取动画实体标签，
        使用 rapidfuzz 的 token_sort_ratio 算法比较英文/日文标题，阈值 >= 0.85 才算匹配。
      - Step 3：剩余未匹配的记录标记为 unmatched。
    匹配结果写入 match_result 表，同时将 Wikidata 实体信息缓存到 wikidata_anime 表。

(4) Export（数据导出 — exporter.py）
    职责：将匹配结果导出为 JSON Lines 格式文件。
    实现：JOIN 三张表（mal_anime、match_result、wikidata_anime），每条记录包含
    mal_metadata（MAL 元数据）、wikidata_metadata（Wikidata 元数据）、match（匹配关系
    和置信度）三个部分。支持按 status 过滤导出（matched/unmatched/review）。

(5) Report（处理报告 — report.py）
    职责：生成本轮运行统计和累计统计报告。
    实现：汇总本轮新增获取数、质量检查通过/未通过数、精确匹配数、模糊匹配数、
    未匹配数，以及累计的总处理条目、匹配率、置信度区间分布等。

编排层（runner.py）：
    负责按顺序调用以上五个阶段，管理 pipeline_stage checkpoint，处理 --skip-fetch、
    --skip-match、--no-export 等跳过选项，并在每个阶段入口设置 checkpoint 标记。

数据库层（database.py）：
    提供 SQLite 连接管理（WAL 模式、外键约束、事务自动 commit/rollback）、
    表初始化（5 张表）、checkpoint 读写、upsert_mal_anime 等 CRUD 操作。

CLI 入口（run.py）：
    使用 argparse 解析命令行参数，支持 --export-only、--report-only 等独立模式。

2. Checkpoint 设计：存什么状态、存在哪、如何保证幂等性

2.1 存在哪

    所有 checkpoint 存储在 SQLite 数据库（data/pipeline.db）的 checkpoint 表中：

    CREATE TABLE checkpoint (
        stage       TEXT PRIMARY KEY,    -- checkpoint 名称
        last_value  TEXT NOT NULL,       -- 进度值
        updated_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    );

    使用 ON CONFLICT(stage) DO UPDATE 实现 upsert，保证同名 checkpoint 只有一条记录。

2.2 存什么状态

    项目中使用了以下 checkpoint：

    (a) fetch（API 模式）
        - stage = "fetch"
        - last_value = 页码数字字符串，如 "5"
        - 含义：已成功获取到第 5 页，下次从第 6 页开始
        - 更新时机：每页数据全部 upsert 到数据库后，在同一个事务中更新 checkpoint

    (b) fetch_csv（CSV 模式）
        - stage = "fetch_csv"
        - last_value = "文件名:行号"，如 "anime-dataset-2023.csv:1000"
        - 含义：该文件已处理到第 1000 行，下次从第 1001 行继续
        - 更新时机：每批（500 条）数据写入数据库后更新，或达到 max_items 限制时更新

    (c) pipeline_stage（阶段级）
        - stage = "pipeline_stage"
        - last_value = "fetch" / "quality" / "match" / "export" / "done"
        - 含义：记录当前 pipeline 正在执行的阶段
        - 更新时机：每个阶段开始时设置，pipeline 全部完成后设为 "done"
        - 注意：当前实现中此 checkpoint 仅作为状态记录，不用于跳过阶段。
          每个阶段都会执行，依靠各自内部的增量/幂等机制实现断点恢复。

    (d) match_stage（匹配子阶段）
        - stage = "match_stage"
        - last_value = "fuzzy" / "done"
        - 含义：精确匹配完成后设为 "fuzzy"，全部匹配完成后设为 "done"
        - 作用：重启后如果 match_stage = "fuzzy"，跳过精确匹配直接进入模糊匹配

    (e) match（匹配进度）
        - stage = "match"
        - last_value = 最大已匹配的 mal_id
        - 更新时机：整个匹配阶段完成后记录

2.3 如何保证幂等性

    各阶段采用不同的幂等策略：

    (a) Fetch 阶段
        - 数据写入使用 upsert_mal_anime()，即 INSERT ... ON CONFLICT(mal_id) DO UPDATE
        - 同一条 mal_id 的记录重复写入只会更新字段值，不会产生重复行
        - checkpoint 确保不会重复读取已处理的数据行
        - 即使在一批数据写入中途崩溃，由于 SQLite 事务保护（get_connection 的
          上下文管理器自动 commit/rollback），要么整批成功要么整批回滚，
          checkpoint 不会超前于实际数据

    (b) Quality Check 阶段
        - 每次执行前先 DELETE 目标 ID 的旧 quality_issues 记录，再重新检查并插入
        - 因此无论执行多少次，结果都一致
        - 这是典型的"先删后插"幂等模式

    (c) Match 阶段
        - match_result 表写入使用 ON CONFLICT DO NOTHING，已有匹配结果不会被覆盖
        - get_unmatched_passed_ids() 查询自动排除已有 match_result 的记录，
          因此重启后只处理尚未匹配的 ID
        - match_stage checkpoint 确保精确匹配完成后不会重跑，直接进入模糊匹配
        - wikidata_anime 缓存表使用 ON CONFLICT(qid) DO UPDATE，重复写入只更新字段

    (d) Export 阶段
        - 每次导出都是全量覆盖写入文件（以 "w" 模式打开），天然幂等

    事务保障：
        get_connection() 上下文管理器确保每次数据库操作在事务中执行：
        - 正常结束 → conn.commit()
        - 异常抛出 → conn.rollback()
        - 数据和 checkpoint 在同一事务中更新，保证一致性
        - SQLite 使用 WAL（Write-Ahead Logging）模式，提升并发读写性能

3. 如何测试了断点恢复功能

3.1 测试环境准备

    (a) 清理旧数据，确保干净的初始状态：
        rm -f data/pipeline.db data/export.jsonl

    (b) 创建 Python 虚拟环境并安装依赖：
        python3 -m venv .venv
        source .venv/bin/activate
        pip install requests aiohttp rapidfuzz

3.2 测试步骤

    Step 1：加载数据（Fetch + Quality，跳过 Match）

        执行命令：
        python run.py --source csv --csv-path data/kaggle --max-items 1000 --skip-match --no-export -v

        结果：
        - 从 CSV 文件成功加载 1000 条数据到 mal_anime 表
        - 质量检查全部通过（1000 passed, 0 failed）
        - checkpoint 记录 fetch_csv = "anime-dataset-2023.csv:1000"

    Step 2：启动匹配阶段并在中途 kill 进程

        执行命令：
        python run.py --skip-fetch --no-export -v &
        PID=$!
        sleep 8
        kill $PID

        模拟的故障：进程在匹配阶段的精确 ID 匹配过程中被 SIGTERM 信号终止。
        此时 pipeline 正在通过 SPARQL 查询 Wikidata，已发起 HTTPS 连接但尚未
        收到完整响应。

        kill 后检查数据库状态：
        - pipeline_stage = "match"（中断在匹配阶段）
        - mal_anime 记录数 = 1000（数据完好，未丢失）
        - match_result 记录数 = 0（匹配结果尚未保存，因为精确匹配还没完成）
        - checkpoint 表保持一致状态

    Step 3：重启 pipeline 验证断点恢复

        执行命令：
        python run.py --skip-fetch --no-export -v

        观察到的日志：
        ── Stage 1: Fetch (skipped) ──                           ← 按参数跳过 fetch
        ── Stage 2: Quality Check ──                             ← 幂等重跑质量检查
        Quality check: 1000 passed, 0 failed                    ← 数据完好
        ── Stage 3: Cross-source Matching ──                     ← 进入匹配阶段
        Matching 1000 MAL IDs…                                   ← 从头开始匹配
        Step 1: Exact ID matching…                               ← 重跑精确匹配

        结果分析：
        - 1000 条数据完好保存，没有丢失
        - Quality check 正常重跑，结果一致
        - Match 阶段重新开始处理所有 1000 条 ID（因为之前的精确匹配还没完成，
          没有任何 match_result 记录，所以 get_unmatched_passed_ids() 返回全部 ID）
        - 符合预期行为：不会从 pipeline 的第一步（fetch）重新开始

    Step 4：验证增量加载

        执行命令：
        python run.py --source csv --csv-path data/kaggle --max-items 1000 --skip-match --no-export -v

        结果：
        - CSV 从第 1001 行继续加载（checkpoint 记录了上次停止位置）
        - 新增 1000 条数据，数据库中总计 2000 条
        - 质量检查对 2000 条记录执行，全部通过

3.3 测试结论

    (a) Fetch 阶段断点恢复 ✓
        checkpoint 精确记录到 CSV 的文件名和行号，重启后从断点继续加载，
        不会重复处理已入库的数据。

    (b) Match 阶段断点恢复 ✓
        - 如果在精确匹配过程中被 kill（match_result 中无记录），重启后重跑精确匹配
          ——数据不丢失，但会有重复的 SPARQL 请求（可接受的代价）
        - 如果在精确匹配完成、模糊匹配开始前被 kill（match_stage = "fuzzy"），
          重启后跳过精确匹配直接进入模糊匹配
        - match_result 使用 ON CONFLICT DO NOTHING，不会产生重复记录

    (c) 数据完整性 ✓
        所有已入库的 mal_anime 数据在 kill 后完好无损，事务机制保证了
        数据和 checkpoint 的一致性。

    (d) 整体行为 ✓
        中断后重启不会从 pipeline 开头重新执行所有阶段，而是每个阶段各自
        通过内部的增量/幂等机制实现"继续处理"的效果。

4. 如果要将这个 Pipeline 部署为长期运行的生产服务，还需要考虑什么

4.1 调度与运行管理

    (a) 定时调度：当前需要手动执行 python run.py，生产环境需要接入调度系统
        （如 cron、Airflow、Prefect、Dagster），实现定期自动运行。
    (b) 进程守护：需要 supervisor、systemd 等进程管理器，确保崩溃后自动重启。
    (c) 并发控制：需要防止多个 pipeline 实例同时运行导致数据冲突，
        可通过文件锁（fcntl.flock）或数据库锁实现。

4.2 数据库升级

    (a) SQLite 局限：SQLite 不支持并发写入、没有用户权限管理、单文件存储不适合
        大规模数据。生产环境应迁移到 PostgreSQL 或 MySQL。
    (b) 连接池：当前每次操作都新建连接，应使用连接池（如 SQLAlchemy）减少开销。
    (c) Schema 迁移：需要引入 Alembic 等工具管理数据库 schema 版本和迁移。
    (d) 索引优化：随着数据量增长，需要根据查询模式添加合适的索引。

4.3 监控与告警

    (a) 结构化日志：当前使用 Python logging，生产环境应输出 JSON 格式日志，
        接入 ELK（Elasticsearch + Logstash + Kibana）或类似日志系统。
    (b) 指标监控：需要暴露关键指标（每轮处理量、匹配率、错误率、各阶段耗时），
        接入 Prometheus + Grafana 等监控系统。
    (c) 告警：匹配率异常下降、连续失败、外部 API 不可用等情况需要触发告警
        （邮件、Slack、PagerDuty 等）。
    (d) 健康检查：提供 HTTP 健康检查端点，方便运维监控服务状态。

4.4 外部 API 稳定性

    (a) 限流策略增强：当前使用固定间隔 + 指数退避，生产环境应实现令牌桶或
        滑动窗口限流，更精确地控制请求速率。
    (b) 熔断机制：当 Wikidata SPARQL 或 Jikan API 持续不可用时，应触发熔断
        而非无限重试，避免浪费资源。
    (c) 缓存层：对 Wikidata 查询结果增加 Redis 缓存层，减少重复请求。
    (d) 多端点/备用源：配置备用 SPARQL 端点或镜像，提高可用性。

4.5 数据质量与治理

    (a) 规则可配置化：当前质量规则硬编码在代码中，应支持通过配置文件或数据库
        动态管理规则，方便业务人员调整。
    (b) 数据血缘：记录每条数据的来源、处理时间、经过的规则，方便问题追溯。
    (c) 人工审查流程：match_result 中 status = "review" 的低置信度记录需要
        提供审查界面或 API，支持人工确认/拒绝。
    (d) 数据过期与清理：长期运行后数据量持续增长，需要制定数据保留策略和归档方案。

4.6 安全性

    (a) 配置管理：API 端点地址、数据库路径等敏感配置应从环境变量或密钥管理服务
        读取，不硬编码在代码中。
    (b) 网络安全：外部 API 请求应通过代理，避免直接暴露服务器 IP。
    (c) 数据备份：SQLite 数据库文件需要定期备份，防止数据丢失。

4.7 可扩展性

    (a) 水平扩展：当前是单进程串行处理，如果数据量增大，需要考虑：
        - Fetch 阶段：多线程/异步并发获取（项目已引入 aiohttp 依赖但尚未使用）
        - Match 阶段：将 ID 分片后并行执行 SPARQL 查询
        - 使用消息队列（如 Redis Queue、Celery）将各阶段解耦为独立 worker
    (b) 更多数据源：当前只支持 MAL → Wikidata，未来可能需要接入 AniList、
        AniDB 等其他数据源，需要设计统一的数据源适配器接口。
    (c) 更多匹配算法：可扩展为基于嵌入向量的语义匹配（如 sentence-transformers），
        提高模糊匹配的准确率。

4.8 测试与 CI/CD

    (a) 单元测试：当前没有测试代码，需要为每个模块编写单元测试。
    (b) 集成测试：使用 mock 外部 API，端到端测试完整 pipeline 流程。
    (c) CI/CD 流水线：代码提交后自动运行测试、代码质量检查（linting）、构建部署。
    (d) 灰度发布：新版本先在小规模数据上验证，确认无误后全量上线。
